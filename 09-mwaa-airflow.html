MWAA and Airflow Workflow Orchestration

Why Workflow Orchestration Exists

Modern data systems are rarely one-step scripts.

They usually require:

  • Ingesting data from multiple systems
  • Validating data quality
  • Transforming raw data into useful models
  • Loading outputs into warehouses or applications
  • Notifying downstream consumers
  • Retrying failures safely

Without orchestration, workflows become fragile collections of scripts, jobs, and manual checks.

Example: A Simple Data Pipeline

This looks simple, but production adds complexity:

  • What if validation fails?
  • What if the warehouse is temporarily unavailable?
  • What if yesterday’s data needs to be reprocessed?
  • What if two tasks can run in parallel?

Orchestration Solves Coordination Problems

Workflow orchestration coordinates:

  1. Task order
  2. Scheduling
  3. Dependencies
  4. Retries
  5. Failure handling
  6. Parallelism
  7. State tracking
  8. Observability

The orchestrator answers:

What should run next, when, and under what conditions?

Orchestration vs Execution

Concept Main responsibility Example
Orchestration Coordinates workflow logic Airflow decides task order
Execution Performs actual work Spark transforms a dataset

Airflow is primarily an orchestrator.

It can run Python code, but it is not designed to be a distributed compute engine for large-scale data processing.

Mental Model

Airflow coordinates the workflow; the compute may happen on workers or in external systems.

What Is Apache Airflow?

Apache Airflow is an open-source platform to:

  • Author workflows as Python code
  • Schedule workflows
  • Manage dependencies
  • Retry failed tasks
  • Monitor task state
  • Backfill historical runs

Airflow models workflows as DAGs.

A DAG is the central abstraction.

DAG Theory

A DAG is a Directed Acyclic Graph.

Term Meaning in Airflow
Directed Edges have direction; order matters
Acyclic No circular dependencies
Graph Tasks connected by dependencies
Node A task
Edge A dependency

Valid DAG

This is valid because dependencies move in one direction and never loop back.

Invalid DAG: Cycle

This is invalid because Task A depends indirectly on itself.

Airflow cannot create a stable execution order from this graph.

Why Acyclic Matters

Acyclic graphs allow Airflow to determine:

  • Which tasks have no unmet dependencies
  • Which tasks can run now
  • Which tasks must wait
  • Which tasks are blocked by failures
  • When the whole DAG run is complete

Cycles make this impossible.

Parallelism from DAG Structure

Tasks B, C, and D can run in parallel after A succeeds.

Parallelism is determined by both:

  • DAG structure
  • Runtime concurrency limits

DAGs Enable Fault Isolation

If one task fails, Airflow can isolate the failure.

If D fails, Airflow does not necessarily need to rerun A, B, and C.

This is one reason task granularity matters.

Airflow Core Concepts

Concept Meaning
DAG Workflow definition
DAG Run One execution of a DAG
Task Unit of work in a DAG
Task Instance One execution of a task in a specific DAG Run
Operator Template for a task action
XCom Small cross-task message
Scheduler Decides what should run
Worker Executes tasks

DAG vs DAG Run

A DAG is the blueprint.

A DAG Run is a specific execution of that blueprint.

Task vs Task Instance

A task is defined once in code.

A task instance is created each time the task runs.

This distinction is essential for debugging Airflow state.

Airflow Architecture

The scheduler decides what should run.

Workers execute tasks.

The metadata database stores the truth about workflow state.

Component: Scheduler

The scheduler is the orchestration brain.

Responsibilities:

  • Parse DAG files
  • Create DAG Runs
  • Evaluate dependencies
  • Queue runnable tasks
  • Respect concurrency limits
  • Update scheduling state

Important mental model:

The scheduler continuously evaluates whether task instances are ready to run.

Component: Metadata Database

The metadata database stores:

  • DAG definitions and serialized metadata
  • DAG Run state
  • Task Instance state
  • Retry history
  • XCom values
  • Connection and variable metadata

Airflow is a state-driven system.

The metadata database is the source of truth.

Component: Workers

Workers execute tasks.

Tasks may:

  • Run Python functions
  • Execute shell commands
  • Call APIs
  • Submit jobs to AWS Glue, EMR, ECS, Lambda, or Spark
  • Read/write to databases or object storage

Workers report state and logs back to Airflow infrastructure.

Component: Webserver / UI

The Airflow UI helps users inspect:

  • DAG list
  • Graph view
  • Grid view
  • Logs
  • Import errors
  • XComs
  • Retries
  • Task duration
  • Connections, variables, and pools

The UI mostly reads state from the metadata database.

Executor Concept

The executor determines how Airflow launches tasks.

Executor Common use
SequentialExecutor Local single-task testing
LocalExecutor Parallel local execution
CeleryExecutor Distributed workers
KubernetesExecutor Task-per-pod execution

MWAA abstracts much of this infrastructure away from the user.

Why MWAA?

Amazon Managed Workflows for Apache Airflow is AWS’s managed Airflow service.

MWAA reduces the need to operate:

  • Scheduler infrastructure
  • Webserver infrastructure
  • Worker infrastructure
  • Metadata database
  • Logging integration
  • Scaling infrastructure
  • AWS networking integration

MWAA Architecture

MWAA connects Airflow to AWS-native infrastructure.

What MWAA Manages

MWAA manages:

  • Airflow scheduler
  • Airflow webserver
  • Worker environment
  • Metadata database infrastructure
  • Logs to CloudWatch
  • Environment scaling integration
  • Airflow environment lifecycle

You still manage:

  • DAG code
  • Dependencies
  • IAM permissions
  • S3 deployment structure
  • VPC connectivity choices
  • Airflow configuration choices

MWAA Deployment Model

The DAG file is deployed to S3, then MWAA parses and schedules it.

MWAA and S3

Typical S3 layout:

s3://your-mwaa-bucket/
  dags/
    example_dag.py
  plugins/
    custom_plugin.zip
  requirements.txt

S3 is the source location for DAGs and environment dependencies.

Common issue:

The DAG is uploaded, but Airflow does not show it because of path, parsing, or dependency errors.

MWAA and IAM

IAM controls what your Airflow environment can access.

Examples:

  • Read/write S3 objects
  • Start AWS Glue jobs
  • Access Secrets Manager
  • Publish CloudWatch logs
  • Query Redshift or Athena

Many MWAA task failures are not Airflow logic failures.

They are permissions failures.

MWAA and VPC

VPC configuration controls network reachability.

Airflow tasks may need access to:

  • Private databases
  • Internal APIs
  • VPC endpoints
  • AWS services
  • Internet endpoints

Common failure:

A task is logically correct but cannot reach the service it needs because of subnet, route table, endpoint, or security group configuration.

Airflow Execution Lifecycle

Use this lifecycle to debug almost every Airflow issue.

Debugging by Lifecycle Stage

Stage Common symptom First place to check
Parse DAG DAG missing Import Errors

Queue task Task stuck queued workers, pools, concurrency
Execute task Task fails task logs
Access AWS service Access denied IAM execution role
Network call Timeout VPC, route, endpoint, security group
